# 数据结构课程个人项目：校园导航

## 1. 项目背景

在现实中，一所大学的校园是由**地点**（教学楼、食堂、宿舍、体育馆、校门等）和**道路**组成的图结构。然而，校园并不是一成不变的：

- 道路可能因施工或活动**临时关闭**；
- 不同用户可能关注**不同的路径指标**（走最短距离还是最快到达）；
- 管理者需要知道哪些地点或道路是**关键设施**——一旦它们不可用，校园的连通性会受到影响。

本项目要求你设计并实现一个**支持动态图更新、路径规划、连通分析和关键节点分析**的校园路线规划系统。

## 2. 项目定位

本项目是**数据结构课程的核心实践环节**，重点考察：

1. **图 ADT 设计**：顶点与边信息建模、邻接表存储、名称索引、动态增删改。
2. **图算法实现**：连通分量分析（BFS/DFS）、Dijkstra 最短路径（含时刻约束变种与必经点路径）、最小生成树（Kruskal 或 Prim 任选）、关键节点与关键边分析。
3. **工程能力**：文件读写、命令解析、异常处理、模块化组织。
4. **复杂度意识**：正确使用邻接表和索引结构，避免低效做法，能在较大规模的图上运行。

## 3. 图模型

### 3.1 顶点（地点）

每个地点包含以下字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| `place_id` | 字符串 | 唯一 ID，如 `P0001`，无空格 |
| `display_name` | 字符串 | 展示名称，无空格 |
| `category` | 字符串 | 类别：`Teaching`、`Dining`、`Dormitory`、`Sports`、`Medical`、`Other` 等 |
| `stay_time` | 整数 | 建议停留时间（分钟） |
| `open_time` | 字符串 | 开放时间，如 `08:00` |
| `close_time` | 字符串 | 关闭时间，如 `22:00` |

### 3.2 边（道路）

每条道路包含以下字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| `from_id` | 字符串 | 起点 place_id |
| `to_id` | 字符串 | 终点 place_id |
| `distance` | 整数 | 距离（米） |
| `walk_time` | 整数 | 步行耗时（分钟） |
| `status` | 字符串 | `open` 或 `closed` |

### 3.3 图的性质

1. 默认是**无向图**。
2. 所有核心算法默认只对 `status=open` 的边生效。
3. 必须支持在图建立后继续进行动态更新（增删改点边、开关道路）。

## 4. 必做功能（75 分）

### 4.1 数据维护（22 分）

1. 从 CSV 文件加载地点和道路数据，并能将当前图数据保存回 CSV 文件。
2. 新增、删除、修改地点。
3. 新增、删除、修改道路。
4. 关闭 / 打开道路。

### 4.2 查询功能（8 分）

1. 按 place_id 查询地点完整信息。
2. 按类别查询所有地点 ID。
3. 查询某地点的所有邻接道路及其详细信息。

### 4.3 图算法（35 分）

#### A. 连通分量分析

计算当前图（仅考虑 open 边）的连通分量个数及每个分量的规模。

#### B. 最短路径（双模式）

支持两种边权模式：

- 按 `distance` 最短（DIST 模式）；
- 按 `walk_time` 最短（TIME 模式）。

输出总代价和完整路径。同一份 Dijkstra 实现应当能处理两种模式（建议把权重选择参数化，不要写两份）。

> 想做更挑战的版本——分层图（K 张加速通行券）的最短路径——见《扩展与加分要求》中的拓展任务 1。

#### B'. 时刻约束最短路径

在给定时刻 `HH:MM` 下，求两点间的最短路径，模式仍是 `DIST` 或 `TIME` 二选一。除了基础最短路径的条件外，额外约束：

- 仅能使用 `status = open` 的道路；
- 路径上**每个**地点（起点、终点、途径点）在该时刻都必须开放
  （即 `open_time ≤ time ≤ close_time`，HH:MM 可用字符串比较直接判断）；
- 若起点或终点在该时刻未开放，视为不可达。

实现上可视为在"当前时刻快照"下过滤掉不可用的点边，再跑一次普通 Dijkstra。

#### C. 必经点路径规划

给定起点、终点和一串**必须按给定顺序经过的地点**，求总路径，模式仍是 `DIST` 或 `TIME` 二选一。

实现思路：将问题拆成若干次单源最短路后拼接。

#### D. 最小生成树

对当前所有 **open** 道路构成的图计算 MST，权值按 `distance` 计算。若图不连通，输出 `DISCONNECTED`。

实现可在 **Kruskal + 并查集** 与 **Prim**（PPT 第 4 章两种算法都讲了）中**任选其一**，或同时实现并对比。报告中需说明选择的理由及该算法在你存储结构上的复杂度。

#### E. 关键节点与关键边分析

输出当前图（仅考虑 open 边）中的**关键节点**和**关键边**：

- **关键节点**：删去该顶点（同时删去其所有相邻边）后，图的**连通分量数增加** ⇒ 该顶点是关键节点；
- **关键边**：删去该边后，图的**连通分量数增加** ⇒ 该边是关键边；

### 4.4 数据结构设计（10 分）

这是本项目作为**数据结构课程作业**的核心考察维度。要求你**自行设计**图及其辅助结构（顶点索引、邻接存储、删除策略等），骨架不规定具体存储方式。评分围绕三个维度：

1. **设计想法**：为什么这样设计？考虑过哪些替代方案？权衡点是什么？
2. **质量**：所选结构是否与操作特性匹配？增删改查的复杂度是否合理？是否能处理边界情况？
3. **兼容性**：同一套结构能否顺畅支撑所有必做功能？扩展到加分项时是否还能复用？

## 5. 程序接口与运行方式

为方便助教用统一测试数据批量评测，所有同学的程序都必须提供 **CLI 批处理入口**，严格按《命令接口规范》中定义的命令名、参数顺序、输出格式实现。

**硬性要求：**

1. 能读取助教提供的 `places.csv` 和 `roads.csv`（字段格式见 §6）；
2. 能用 `./CampusNavigation < command.txt > answer.txt` 方式跑通批处理；
3. 命令名、关键字（`OK` / `ERROR` / `PATH` / `MST` / `CRITICAL` 等）、字段顺序、排序规则**严格符合**《命令接口规范》。

**请注意：**

- 这一要求**只针对评测接口**。你的程序**内部**如何组织代码、用什么数据结构、要不要在 CLI 之外再加一个 REPL/菜单/Web 界面，**完全自由**。
- 如果你做了图形化界面（拓展任务 3），CLI 评测接口仍需保留——图形界面是**额外**做的，不能替代 CLI。

## 8. 代码骨架说明

项目提供了一份代码骨架**仅作参考**，包括：

| 文件 | 内容 |
|------|------|
| `LocationInfo.h/cpp` | 地点信息结构体 |
| `LGraph.h/cpp` | 图 ADT 的公共接口（不规定内部存储，需自行设计） |
| `Algorithm.h/cpp` | 图算法接口（含 TODO） |
| `CsvIO.h/cpp` | CSV 读写模块（含 TODO） |
| `CommandProcessor.h/cpp` | 命令解析与分发（含 TODO） |
| `main.cpp` | 主程序入口 |
| `GraphException.h` | 异常类 |
| `CMakeLists.txt` | 构建配置 |

**骨架完全可选**——你可以：

- **照着骨架走**：实现每个 `TODO`，自行设计 `LGraph` 内部数据结构；
- **完全重写**：丢弃这份骨架，按你自己的代码风格、模块划分来组织项目；
- **半选半改**：保留骨架的某些部分，重写其余部分。

**唯一硬性约束**（与骨架无关）：

1. 程序必须提供符合《命令接口规范》的 CLI 批处理入口（见 §5）；
2. 必须用 C++17 标准编译通过；
3. 不能调用外部图算法库，所有图算法必须自行实现。
