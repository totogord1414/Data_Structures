# 基于 Huffman 编码的压缩系统 - 实验报告

## 一、自定义哈希算法与一致性校验分析

### 1. 算法简介
本项目实现了两种原理截然不同的自定义散列算法，用于在解压后对文件完整性进行校验：
* *** 算法1**：利用素数 `5381` 作为初始值，通过 `(hash << 5) + hash` 替代乘以 33，辅以加法运算。算法极简，速度极快，对英文等常规文本文件具有极佳的散列分布。
* **算法2**：使用了底数 `131` 和大素数模数 `1000000007`，以多项式展开的形式将字节流转换为哈希值，并在计算过程中手动取模防止溢出。

### 2. 碰撞概率估算（生日悖论分析）
哈希算法将任意长度的文件映射为一个固定长度的整数，因此必定存在“碰撞”（两个不同文件哈希值相同）。
* 根据**生日悖论**，对于一个 64 位无取模的哈希函数，哈希空间为 $2^{64}$。大约需要计算 $\sqrt{2^{64}} = 2^{32}$（约 42 亿）次才会有 50% 的概率发生碰撞。
* 在本项目中，虽然多项式哈希通过 `% 1000000007` 将空间压缩到了约 10 亿，但由于本项目的测试文件最大仅为 21MB（如《完美世界.txt》），远未达到此量级，因此在本项目场景下，发生哈希碰撞的概率无限趋近于 0，用来做完整性校验完全可靠。

### 3. MD5 (128位) 与 64位哈希的安全性对比
* **64位哈希（如本项目）**：通常是非加密哈希，主要目的是“防误改”（如网络传输错误或磁盘损坏），其计算速度极快。但 $2^{64}$ 的空间在现代 CPU 算力下极易被暴力破解，黑客可以轻易构造出内容不同但 64 位哈希值相同的恶意文件。
* **MD5（128位）**：具有 $2^{128}$ 的巨大哈希空间，需要高达 $2^{64}$ 次尝试才会大概率发生碰撞。在实际工程中，128 位哈希能够提供基础的“防篡改”安全性，使得恶意构造碰撞的成本呈指数级上升。

### 4. 哈希比较 vs 逐字节比较 (`diff` / `fc`)
* **哈希比较**：
  * **优点**：内存开销极小。只需在压缩/解压过程中顺带计算出一个整数。只需比对一个几十个字节的数字即可验证上 GB 文件的完整性。
  * **缺点**：理论上存在碰撞风险；若文件不一致，无法指出具体是哪个字节出错了。
* **逐字节比较 (fc/diff)**：
  * **优点**：100% 绝对准确，不存在碰撞问题；能够精确定位差异发生的位置。
  * **缺点**：磁盘 IO 极高，必须将两个文件同时存放在本地磁盘上进行双路读取；无法用于验证“已经不在本地”的文件（如验证刚下载的文件与云端文件是否一致）。

## 二、信息熵与编码效率分析

### 1. 计算
根据香农信息熵公式：$H = -\sum p_i \log_2(p_i)$ ，信息熵代表了对该文件进行无损压缩的**理论极限（即最短的平均码长）**。同时，哈夫曼编码的实际平均码长公式为 $L = \sum p_i \times len(code_i)$。

**计算过程与脚本说明**：
我用**AI编写了 Python 测试脚本 (`calc_entropy.py`)** 来完成自动化测算。该脚本的核心逻辑如下：
1. **频率统计**：以二进制 (`rb`) 模式读取全文件，使用 `collections.Counter` 统计全部 256 种字节的绝对频次，并除以总字节数得到真实的概率 $p_i$。
2. **理论信息熵计算**：遍历频率表，严格根据数学公式累加 $-p_i \log_2(p_i)$ 得到理论极值 $H$。
3. **真实码长模拟**：使用 Python 的优先队列 (`heapq`) 完全模拟了 C++ 主程序中的哈夫曼建树过程，精确求出了每一个字节对应的叶子节点深度（即真实编码长度 $len_i$），并加权平均得出 $L$。

通过该脚本对全文件的每一个字节进行频率扫描和测算，我们得到了如下结果：

#### 测试一：`hamlet.txt` (纯英文文本, 约180KB)
* **香农信息熵 ($H$)**：计算得 **4.8492** bits/char
* **实际哈夫曼平均码长 ($L$)**：计算得 **4.8828** bits/char
* **编码效率 ($\eta = H / L$)**：**99.31%**
* **分析**：这篇莎士比亚的纯英文原著，其字符主要集中在 26 个小写字母、极度高频的空格以及少量标点符号上。字母频率呈现出极其陡峭的长尾分布（例如字母 'e'、't' 极多，而 'z'、'q' 极少）。这种极度“不均匀”的分布导致文件的整体**不确定度非常低**，因此其信息熵低（仅 4.8492，远小于单字节的 8 bits）。
哈夫曼算法在这里为超高频字符分配了短至 2-3 位的编码，最终的平均码长（4.8828）极度逼近理论极限。

#### 测试二：`斗破苍穹.txt` (中英混合大文件, 约16MB)
* **香农信息熵 ($H$)**：计算得 **5.8239** bits/char
* **实际哈夫曼平均码长 ($L$)**：计算得 **5.8576** bits/char
* **编码效率 ($\eta = H / L$)**：**99.43%**
* **分析**：中文字符通常采用 UTF-8 编码，每个汉字在底层占用 3 个字节。在高达 16MB 的海量中文字符叠加下，UTF-8 编码高位字节和控制字节的频率分布变得比纯 ASCII 更加分散且均匀。
随着字节分布逐渐趋于“均匀”，文件的“系统随机性（信息熵）”随之升高，达到了 5.8239 bits。这也导致在构建 Huffman 树时，树的形态变得相对**扁平化**，大部分叶子节点的深度非常接近。因为无法再依靠极少数的超短编码来抵消大盘字符的开销，所以最终的平均码长上升到了 5.8576 bits。

### 2. 编码效率 ($\eta$) 与 压缩率 对比分析
(报告要求的编码效率是不是有问题，两者在这一层面上的差距不大)
哈夫曼编码的平均码长 $L$ 会不断逼近信息熵 $H$。理论上的“哈夫曼编码效率”定义为 $\eta = H / L \times 100\%$。

对比 `hamlet.txt` 和 `斗破苍穹.txt`，我们发现了一个非常重要的结论：
1. **哈夫曼编码效率 ($\eta$) 差异极小**：
   纯英文文本的 $\eta$ 为 **99.31%**，中英混合大文件的 $\eta$ 为 **99.43%**。两者的哈夫曼编码效率**几乎没有差异**，都极其逼近 100%。这在理论上证明了：无论什么语言、什么分布，哈夫曼编码作为最优的无损前缀编码，都能在给定的概率分布下榨干最后一点冗余，极其完美地贴合信息熵。
2. **实际压缩率 (绝对平均码长 $L$) 差异巨大**：
   虽然 $\eta$ 差不多，但纯英文能被压到平均 **4.88** bits/char（压缩率约 61%），而中文只能被压到 **5.85** bits/char（压缩率约 73%）。两者的**绝对压缩效率**相差整整 1 个 bit。

**产生这种绝对压缩差异的根本原因与字节分布直接相关：**
* 对于 **`hamlet.txt` (纯英文)**：其字符种类极少（主要是 26 个字母和空格），且频率分布呈现出**极端的长尾（不均匀）**分布。比如空格的出现概率高达 14.7%。这种极度倾斜的分布导致系统“不确定度”极低，其基础信息熵 $H$ 就极低。哈夫曼树的高低落差非常大，能为高频字符分配短至 3 位的编码，从而拉低了整体的绝对码长。
* 对于 **`斗破苍穹.txt` (中英混合)**：中文字符采用 UTF-8 编码，每个汉字占 3 个字节。在高达 16MB 的海量中文字符叠加下，256 种可能的字节被更加**均匀**地利用。分布越均匀，系统的“随机性（信息熵 $H$）”就越大（达到了 5.82）。这导致构建出的 Huffman 树形态更加**扁平化**，大部分叶子节点的深度非常接近（都在 5~7 左右），没有绝对压倒性的超高频字节来拉低平均值，因此绝对码长更长。
