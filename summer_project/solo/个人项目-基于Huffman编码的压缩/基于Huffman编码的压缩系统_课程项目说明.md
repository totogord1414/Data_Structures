## 一、项目背景

### 1.1 什么是 Huffman 编码
Huffman 编码是一种经典的无损数据压缩算法，核心思想很简单：
> 出现频率高的字符用短编码，出现频率低的字符用长编码，这样整体占用的空间就变小了。

举个例子：假设一个文件里字母 a 出现了 1000 次，字母 z 只出现了 3 次。如果给 a 分配编码 01（2 位），给 z 分配编码 110101（6 位），相比每个字符都用 8 位（1 字节）存储，能节省大量空间。

### 1.2 Huffman 编码是怎么工作的
整个过程分四步：
1. **统计频率** — 扫描文件，统计每个字节（0x00-0xFF，共 256 种）出现了多少次。
2. **建树** — 把每个出现的字节作为一个"叶子节点"（权重=出现次数），反复取出权重最小的两个节点合并成一个新节点（权重=两者之和），直到只剩一个根节点。这就是 Huffman 树。
3. **生成编码** — 从根节点出发，走左边记 0，走右边记 1，到达某个叶子节点时，路径上的 01 串就是这个字节的 Huffman 编码。
4. **压缩 / 解压** — 编码时把每个字节替换成对应的 01 串写入文件；解压时从根节点出发，根据 01 串走到叶子节点还原字节。

> 直观理解：Huffman 树就像一张"翻译表"，把原始字节翻译成更短的 01 串，解压时再翻译回去。

### 1.3 你要做什么
从零实现一个命令行压缩工具：输入一个文件，输出压缩后的 `.huff` 文件；也能把 `.huff` 文件还原回原始文件。

| 完美世界.txt | 21 MB | 中文小说（纯中文为主） | UTF-8 |

**关于 UTF-8 编码**
* 英文在 UTF-8 下和 ASCII 完全一样，每个字符占 **1 字节**。
* 中文在 UTF-8 下是变长的，一个常用汉字占 **3 字节**。
* **处理方式：按字节处理即可**。 把每个字节当成一个 0-255 的数值，用 `unsigned char` 读取，用一个 `count[256]` 数组统计每种字节出现了多少次。不管是英文还是中文，这个方法都能统一处理，不需要区分字符类型。

**参考代码思路（来自 `中文处理（仅供参考）.md`）：**
```c
FILE* fp = fopen("test.txt", "rb");
unsigned char readbuff[1024];
int count[256] = {0};

while (true) {
    int read_size = fread(readbuff, 1, 1024, fp);
    if (read_size == 0) break;
    for (int i = 0; i < read_size; ++i) {
        count[readbuff[i]]++; // 用字节值做下标直接计数
    }
}
```

## 三、你的程序怎么运行（接口规范）
> **这是最重要的部分。** 你的程序必须严格按以下格式接受命令，因为助教会用脚本进行批量自动测试。格式不对会被视为功能未完成。

你的程序编译后生成的可执行文件命名为 `Huffman`（Windows 下为 `Huffman.exe`）。

### 3.1 三种运行模式

**模式一：压缩**
```
Huffman -c <文件路径>
```
示例：
```
Huffman -c ./data/hamlet.txt
```
读取指定文件，统计频率、构建 Huffman 树、压缩，在同目录下生成以下 3 个文件：

| 生成的文件 | 命名规则 | 内容说明 |
| --- | --- | --- |
| 压缩文件 | 原文件名.huff | 二进制压缩文件 |
| 频率表 | 原文件名_freq.txt | 文本文件，记录每种字节的出现频率 |
| 解压验证文件 | 原文件名_unzip.txt | 压缩后立刻解压还原，用于验证正确性 |

即 `Huffman -c ./data/hamlet.txt` 会在 `./data/` 下生成：
* `hamlet.huff`
* `hamlet_freq.txt`
* `hamlet_unzip.txt`


**模式二：解压**
```
Huffman -x <文件路径>
```
示例：
```
Huffman -x ./data/hamlet.huff
```
读取 `.huff` 压缩文件，解析、重建 Huffman 树、解码，在同目录下生成：

| 生成的文件 | 命名规则 | 内容说明 |
| --- | --- | --- |
| 解压文件 | 原文件名_unzip.txt | 还原后的原始文本 |

即 `Huffman -x ./data/hamlet.huff` 会在 `./data/` 下生成 `hamlet_unzip.txt`。


**模式三：自适应流式压缩**
```
Huffman -a <目录路径>
```
示例：
```
Huffman -a ./data/adaptive_data/
```
模拟"数据流逐步到达"的场景。程序自动处理指定目录下的 `1.txt`、`2.txt`、`3.txt`（按文件名排序），每处理一个新文件，都会累积前面的频率来更新 Huffman 树。**一次运行**完成全部三个阶段。

| 阶段 | 处理的文件 | Huffman 树依据 | 在同目录下生成的文件 |
| --- | --- | --- | --- |
| 一 | 1.txt | 1.txt 自身的频率 | 1_adp.huff、1_freq.txt、1_unzip.txt |
| 二 | 2.txt | 1.txt + 2.txt 的累积频率 | 2_adp.huff、2_freq.txt、2_unzip.txt |
| 三 | 3.txt | 1.txt + 2.txt + 3.txt 的累积频率 | 3_adp.huff、3_freq.txt、3_unzip.txt |

总共生成 **9 个文件**。

> **为什么这样做？** 真实场景（如网络传输）中数据是逐步到达的，编码表需要根据新数据动态更新。阶段二用 1+2 的累积频率建树来压缩 2.txt，是因为此时实际"见过"的数据是 1 和 2 的并集。

### 3.2 接口速查表

| 命令 | 作用 | 生成什么 |
| --- | --- | --- |
| `Huffman -c <file>` | 压缩文件 | `<file>.huff`、`<file>_freq.txt`、`<file>_unzip.txt` |
| `Huffman -x <file>` | 解压文件 | `<file>_unzip.txt` |
| `Huffman -a <dir>` | 自适应流式压缩 | 9 个文件（见 3.1 模式三） |

> **扣分点**：文件名不对、输出位置不对 → 此项不得分。

### 4.2 hamlet.txt 压缩 + 优先队列优化（25 分）
**你要做的：**
* 用 `Huffman -c` 压缩 `hamlet.txt`，确保解压后完全一致。
* 构建 Huffman 树时，**必须使用优先队列（小根堆）**来找频率最小的两个节点，不能用每次线性扫描的方式。

> **为什么？** 如果每次都用 for 循环扫描整个数组找最小值（O(n²) 复杂度），hamlet.txt 还勉强能跑，但后面的 16MB+ 中文文件就非常慢了。用堆可以把构建时间降到 O(n log n)。

**C++ 提示**：用 `std::priority_queue` 配合 `std::greater` 改成小顶堆，或用 `std::multiset`。

> **扣分点**：没用优先队列 / 堆优化 → 此项最多得 10 分。

### 4.3 自定义哈希校验文件一致性（20 分）
**你要做的：**
* 实现**两种不同的自定义散列函数**，用于计算文件的哈希值。
* 用这两种哈希函数分别验证压缩前后文件是否一致（原始文件和 `_unzip.txt` 的哈希值应该完全相同）。
* 在报告中完成分析。

> **重要**：必须自己实现散列函数，不能用 `std::hash` 包装一下交差。

### 4.5 自适应流式压缩（20 分）
**你要做的：**
* 实现 `Huffman -a <目录>` 命令。
* 程序一次运行，依次处理目录下的 `1.txt`、`2.txt`、`3.txt`。
* **关键是**：处理 `2.txt` 时基于 `1.txt + 2.txt` 的累积频率建树，处理 `3.txt` 基于 `1+2+3` 的累积频率建树。

> **直观理解**：就像看视频——视频数据逐步到达，编码器需要根据"目前为止收到的所有数据"来动态调整压缩策略。

`_freq.txt` **说明**：自适应模式下生成的 `_freq.txt` 体现的是**累积频率**（而非当前单个文件的频率），格式和普通模式完全一样。

### 4.6 信息熵与编码效率分析（5 分）
**你要做的：**
* 在报告中计算每个文件的 Shannon 信息熵和 Huffman 平均码长。
* 对比 `hamlet.txt`（纯英文）和 `斗破苍穹.txt`（中英混合）的编码效率，分析差异原因。

**公式：**
* **信息熵：H = −Σ( pᵢ × log₂(pᵢ) )**
  pᵢ = 字节 i 出现的概率。信息熵是理论上能达到的最短平均码长（压缩的理论极限）。
* **平均码长：L = Σ( pᵢ × len(codeᵢ) )**
  codeᵢ = 字节 i 的 Huffman 编码。这是你的 Huffman 编码实际达到的平均码长。
* **编码效率：η = H / L × 100%**
  越接近 100% 说明你的 Huffman 编码越接近理论最优。

**报告中要分析**：为什么中英混合文本的编码效率和纯英文不同？和字节分布有什么关系？
